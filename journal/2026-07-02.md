## Milestone 4.2 — Assistant chat UI + per-client rate limit

**What was built:** (A) A per-IP rate limit on the public `POST /api/assistant/ask` using ASP.NET Core's built-in rate limiter (no NuGet, no background worker) — a named fixed-window policy "assistant", 20 requests/minute, partitioned by client IP, applied only to `AssistantController` via `[EnableRateLimiting("assistant")]`. (B) A plain HTML/Bootstrap/jQuery chat page `wwwroot/assistant.html`: an identity form (username/email/phone) that gates entry to a chat panel, which POSTs questions to our own endpoint and renders the answer with its source documents. Plus a link to it from `index.html`.

**Key decisions:**
- Rate limit is **per-IP + in-memory**: fair per-caller throttle, correct for a single Render instance; the 1-minute window IS the cooldown (`QueueLimit = 0`, reject immediately). Client IP read from the leftmost `X-Forwarded-For` (Render proxy) falling back to `RemoteIpAddress`. `OnRejected` returns a safe 429 JSON message + `Retry-After` header.
- `UseRateLimiter()` sits after `UseAuthorization()` and before `MapControllers()`.
- The identity form is a **soft UX gate, not a security boundary** — identity lives in JS memory only, never persisted, never sent. The IP rate limit is the real control.
- The UI calls **only** `/api/assistant/ask` (RAG key stays server-side) and renders `answer` + de-duplicated source badges. **Verified live** that the source field on the wire is `s.source` (not `document`/camelCase) — `[JsonPropertyName("source")]` governs serialization too.

**Files added/changed:** `Api/Program.cs` (AddRateLimiter + UseRateLimiter), `Api/Controllers/AssistantController.cs` (`[EnableRateLimiting]`), `Api/wwwroot/assistant.html` (new), `Api/wwwroot/index.html` (link added). No DTOs, `RagClient`, or the M4.1 contract touched. No DB writes / no chat-history storage.

**To verify:** `dotnet run --project ShipmentTracking.Api`; fire 21 POSTs/min from one client → first 20 = 200 (or 400 for empty body), 21st = 429 with `Retry-After: 60` (confirmed this session on port 5068). Open `/assistant.html`, fill the identity form, ask "what does CUS-HOLD mean?" → answer with source badges; spam Send → graceful 429 message.

**Remember next session:** Chat is stateless — the DB migration for storing conversations is deliberately deferred. Rate-limit counters reset on app restart (in-process, single instance); fine for Render's single F1 instance.

## Milestone 4.3 — Floating chat widget

**What was built:** Re-packaged the standalone `assistant.html` chat into a reusable floating widget. `wwwroot/assistant-widget.html` is a self-contained fragment (markup + scoped `<style>` + IIFE `<script>`, no library tags) that injects a circular 💬 button fixed to the bottom-right corner; clicking it opens a chat panel via a CSS `transform:scale() + opacity` animation anchored at the bottom-right. The full M4.2 chat logic (identity gate, `$.ajax` to `/api/assistant/ask`, spinner, `s.source` source badges, 429/502/503 error mapping) was re-homed into the IIFE unchanged. `assistant.html` was rewritten as a thin host page that mounts the widget. `index.html` mounts it too via `$(function(){ $('#assistant-widget-mount').load('/assistant-widget.html'); })`. No backend changes.

**Key decisions:**
- **One fragment, no library re-includes:** the widget relies on Bootstrap + jQuery already on the host page (they always are). Re-including jQuery would double-load it.
- **IIFE + `asst-` prefixed IDs:** prevents clobbering the host page's `esc`/`showAlert` globals and prevents duplicate DOM IDs. Both `index.html` and `admin.html` define their own `esc` — a naïve inject would break them.
- **Pure-CSS animation:** toggling `.asst-open` drives `transform:scale(.85→1)` + `opacity(0→1)` with `transition:.18s ease`; `transform-origin: bottom right` makes it grow out of the button corner. No JS animation library.
- **`.load()` without a selector suffix:** jQuery injects the HTML fragment AND runs its inline `<script>` (via `globalEval`). With a selector suffix it would strip the scripts. Mount div must exist in the DOM before `.load()` fires — solved by wrapping in `$(function(){…})`.

**Files added/changed:** `Api/wwwroot/assistant-widget.html` (new), `Api/wwwroot/assistant.html` (rewritten as thin host), `Api/wwwroot/index.html` (mount div + load call; old text link removed). No backend files touched.

**To verify:** `dotnet run`, open `/` → 💬 button bottom-right → click → expands smoothly → identity gate → chat with source badges → close collapses. Host page tracking box still works (proves no globals collision). Same two include lines on any page adds the button there.

## Hotfix — widget rendered as raw text (`</script>` inside JS comment)

**Root cause:** `index.html`'s inline `<script>` block contained a JS `//` comment that included the literal text `</script>`. The HTML tokeniser doesn't understand JavaScript comments — it terminates a `<script>` element at the first literal `</script>` token it encounters, regardless of context. That closed the script element prematurely, ejecting the real `.load()` call and the actual `</script>` into the page body, where they were rendered as text and never executed.

**Fix:** Removed the HTML-tag example lines from the JS comment (one-line change). Moved the copy-paste snippet to the `<!-- … -->` HTML comment beside the mount `<div>`, where `</script>` is completely inert. The `.load()` call, the mount `<div>`, jQuery load order, and all widget logic are unchanged.
