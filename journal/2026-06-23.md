# 2026-06-23 — feature/dapper-tracking

**What was built:** Added the public tracking endpoint (`GET /api/track/{trackingNumber}`) backed by two parameterized Dapper queries (no EF). New files: `PublicTrackingResponse`, `PublicTrackingHistoryItem` DTOs (both in Core), `IShipmentQueries` interface (Core), `ShipmentQueries` implementation (Infrastructure/Queries), and `TrackingController` (Api). Dapper 2.1.79 added to Infrastructure; `SqlConnection` resolved transitively.

**Key decisions:** Two queries over multi-mapping JOIN — far more readable; one extra round-trip is acceptable on a low-traffic public read. Private `ShipmentRow` nested class holds the internal `Id` for query 2 without ever surfacing it in the response. `SenderName`/`ReceiverName` deliberately excluded from the SQL SELECT — a customer's tracking lookup should not reveal the counterparty's personal details.

**Dapper vs EF Core:** EF is an ORM — it tracks changes to objects, builds SQL from LINQ, and manages a unit of work. Dapper is a thin micro-ORM — you write the SQL yourself and Dapper maps result columns to C# properties by name. For the public read, we never need change-tracking; raw SQL is faster and more explicit about exactly what data is read.

**Auth split:** `TrackingController` has no `[ApiKeyAuthFilter]` attribute — it is fully public. `ShipmentsController` still requires the header for every admin action. The two controllers are separate on purpose so the auth boundary is visible and unambiguous.

## Scaling notes & deferred decisions (low-traffic assumptions)

**1. Two round-trips per lookup (shipment query + history query)**
Current state: two separate parameterized queries — one for the shipment row, one for its history. Chosen for readability; the extra round-trip is negligible at low request volume.
High traffic: replace with a single JOIN query + Dapper multi-mapping (`splitOn: "Id"`), eliminating one database round-trip per request.

**2. No caching**
Current state: every `GET /api/track/{trackingNumber}` hits the database directly, even if the same tracking number is looked up repeatedly.
High traffic: add an in-memory cache (`IMemoryCache`) or distributed cache (Redis) keyed by tracking number with a short TTL (e.g. 30–60 seconds). This is the single highest-impact change for a read-heavy public endpoint — a shipment's status changes rarely, so cached reads can absorb most traffic without touching the DB.

**3. Indexing**
Current state: a unique index on `Shipments.TrackingNumber` and a foreign-key index on `StatusHistories.ShipmentId` already exist (created by `InitialCreate`). Both lookup queries hit indexed columns.
High traffic: run `SET STATISTICS IO ON` and inspect query plans under realistic data volume before adding anything. Composite or covering indexes (e.g. including `Status`, `Origin`, `Destination` on the TrackingNumber index) can eliminate key lookups, but add them only when a measured plan shows they're needed.

**4. No rate limiting on the public (key-less) endpoint**
Current state: `GET /api/track/{trackingNumber}` is fully anonymous — a single client or a scraper can hammer it without restriction.
High traffic or abuse: add ASP.NET Core's built-in rate limiter (`AddRateLimiter` / `UseRateLimiter`, introduced in .NET 7) with a fixed-window or sliding-window policy per IP address. No third-party package needed.

**5. Single instance / horizontal scaling**
Current state: the API is completely stateless — all state lives in SQL Server. `SqlConnection` objects are not reused across requests; ADO.NET connection pooling handles the underlying TCP connections transparently.
High traffic: run multiple instances behind a load balancer (e.g. Azure App Service scale-out or Render's horizontal scaling). No sticky sessions or shared in-process state to worry about. If a distributed cache (point 2) is added, use Redis so all instances share the same cache layer.

**Principle:** measure first — run a load test (e.g. k6 or `dotnet-load-test`) and profile with SQL Server's Query Store or Application Insights before optimizing anything. Optimize the proven bottleneck, not the assumed one.

# 2026-06-23 — feature/frontend

**What was built:** Two static HTML pages in `wwwroot/` — `index.html` (public tracking page, no key) and `admin.html` (admin panel with create/list/update-status). Both use Bootstrap 5 (jsDelivr CDN) + jQuery (code.jquery.com). `Program.cs` gained two lines: `app.UseDefaultFiles()` then `app.UseStaticFiles()`, enabling `/` → `index.html` and `/admin.html` serving. No other backend files were touched.

**Key decisions:** Same-origin serving (pages come from the same host as `/api/...`) means the browser never issues a cross-origin request — no CORS config needed, ever. Admin API key is read from a password-type input at runtime by `apiHeaders()`; it is never stored, never hardcoded, never committed. The 400 response from `PUT /status` is plain text (`BadRequest(ex.Message)` in C#), so JS reads `jqXHR.responseText` — not `jqXHR.responseJSON` — and shows the server's exact transition error message verbatim.

**JS structure:** Both pages follow the same pattern: small named functions (`trackShipment`, `createShipment`, `listShipments`, `updateStatus`), shared helpers (`esc()` for HTML-escaping, `formatDate()`, `statusBadgeClass()`), jQuery `$.ajax` with `.done`/`.fail` handlers that branch on `jqXHR.status`. No global state; no frameworks.

**Security notes:** `esc()` HTML-escapes all server strings before `innerHTML` injection, preventing accidental markup injection from tracking-number or name fields. The `apiKey` field has `autocomplete="off"` and `type="password"` to discourage browser storage.
