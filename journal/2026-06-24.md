# 2026-06-24 — feature/deploy

**What was built:** Docker-based deployment to Render's free tier. Added `Microsoft.EntityFrameworkCore.Sqlite` to Infrastructure. `Program.cs` now reads `DatabaseProvider` from config: if `"Sqlite"` it calls `UseSqlite`, otherwise `UseSqlServer` (LocalDB, unchanged locally). Startup calls `db.Database.Migrate()` so the SQLite file and schema are created automatically on first container run. Kestrel binds to Render's injected `PORT` env var when present, falling back to launchSettings locally. `ShipmentQueries.cs` (Dapper) now has a `CreateConnection()` factory that returns `SqliteConnection` or `SqlConnection` based on the same `DatabaseProvider` setting — the SQL is identical for both. New files: `Dockerfile` (multi-stage .NET 10 build + runtime), `.dockerignore`, `DEPLOY.md`.

**Key decisions:** Single migration set works for both providers — SQLite ignores the `SqlServer:Identity` annotation and applies type affinity to SQL Server type strings. `Database.Migrate()` on startup is the simplest approach for a demo: no manual `dotnet ef database update` step in CI/CD. Provider selection is a plain `if/else` in `Program.cs` on a `DatabaseProvider` config value — readable and easy to explain. The Dapper public read endpoint (which bypasses EF) also needed a provider-aware switch; `DbConnection` (the base class) lets Dapper work with either driver without changing the SQL.

**Secrets flow:** `ApiKey` stays out of the repo. Locally it comes from `dotnet user-secrets`. On Render it's a dashboard environment variable. ASP.NET Core's default `IConfiguration` chain (appsettings → env vars → user-secrets) picks it up in both cases with zero code change in `ApiKeyAuthFilter`. `ConnectionStrings__DefaultConnection` uses `__` double-underscore because Render env vars are flat strings and ASP.NET Core maps `__` to `:` for nested config keys.

**NU1903 advisory:** `Microsoft.EntityFrameworkCore.Sqlite` 10.0.9 pulls in `SQLitePCLRaw.lib.e_sqlite3` 2.1.11 (transitive), which has a known high-severity NuGet audit advisory. Build is 0 errors; this is a transitive dependency warning acceptable for a demo. If this were production, add a direct `<PackageReference>` to the patched SQLitePCLRaw version to override the transitive pin.

**Next session:** Push the branch to GitHub, create the Render Web Service, set the three env vars (`DatabaseProvider`, `ConnectionStrings__DefaultConnection`, `ApiKey`) in the Render dashboard, wait for the first deploy, and smoke-test the live URL end-to-end (public tracking page, admin panel, 401 check, illegal transition 400 check).

# 2026-06-24 — feature/auth

**What was built:** Replaced the static `X-Api-Key` filter with JWT-based login and role authorization. New entity `AppUser` (Id, Username, PasswordHash, Role) with a `Users` table in both SQL Server and SQLite migration sets (`AddUsers`). `AuthService` verifies credentials with `PasswordHasher<AppUser>`, issues a signed JWT (HMAC-SHA256, 8 h expiry) carrying sub/username/role claims. `AuthController` exposes `POST /api/auth/login` (anonymous). `ShipmentsController` now has `[Authorize(Roles = "Admin")]`. `AdminUserSeeder` creates one admin on startup if the table is empty and seed config is present. `ApiKeyAuthFilter.cs` deleted. New `login.html`; `admin.html` updated to check JWT on load, attach `Authorization: Bearer` header, redirect to login on 401, and add a Logout button.

**Key decisions:** `PasswordHasher<AppUser>` from `Microsoft.Extensions.Identity.Core` only — no full ASP.NET Identity framework/schema. `JwtSettings` is a plain POCO singleton (no `IOptions<T>`) to keep `AuthService` simple. Login returns the **same 401** for unknown user and wrong password — callers cannot probe which usernames exist. Token stored in `sessionStorage` (cleared on tab close; better for shared machines than `localStorage`). `ClockSkew = TimeSpan.Zero` so 8 h means exactly 8 h. `AppDbContextFactory` added to Infrastructure so `dotnet ef migrations add` works without `Program.cs` needing `Jwt:Key`.

**Render env vars going forward:** `Jwt__Key` (new, required), `Seed__AdminUsername` + `Seed__AdminPassword` (new, required for initial login). **Obsolete — remove from Render:** `ApiKey`.

**Migration commands run:** `dotnet ef migrations add AddUsers -p ShipmentTracking.Infrastructure -s ShipmentTracking.Api` (SQL Server) and `dotnet ef migrations add AddUsers -p ShipmentTracking.Migrations.Sqlite -s ShipmentTracking.Migrations.Sqlite` (SQLite).

**Next session:** Set user-secrets (`Jwt:Key`, `Seed:AdminUsername`, `Seed:AdminPassword`) locally, apply migrations (`dotnet ef database update -s ShipmentTracking.Api`), and smoke-test: login → JWT → admin CRUD → Logout; wrong password → same 401; public tracking page still anonymous.
